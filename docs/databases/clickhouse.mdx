# ClickHouse Connections

TablePro supports ClickHouse via its HTTP interface. ClickHouse is a column-oriented OLAP database built for real-time analytics on large datasets. TablePro connects over HTTP (port 8123 by default), not the native TCP protocol.

## Install Plugin

The ClickHouse driver is available as a downloadable plugin. When you select ClickHouse in the connection form, TablePro will prompt you to install it automatically. You can also install it manually:

1. Open **Settings** > **Plugins** > **Browse**
2. Find **ClickHouse Driver** and click **Install**
3. The plugin downloads and loads immediately - no restart needed

## Quick Setup

<Steps>
  <Step title="Open Connection Form">
    Click **New Connection** from the Welcome screen or **File** > **New Connection**
  </Step>
  <Step title="Select ClickHouse">
    Choose **ClickHouse** from the database type selector
  </Step>
  <Step title="Enter Connection Details">
    Fill in host, port, username, password, and database name
  </Step>
  <Step title="Test and Connect">
    Click **Test Connection**, then **Create**
  </Step>
</Steps>

## Connection Settings

### Required Fields

| Field | Description | Default |
|-------|-------------|---------|
| **Name** | Connection identifier | - |
| **Host** | Server hostname or IP address | `localhost` |
| **Port** | ClickHouse HTTP port | `8123` |

### Optional Fields

| Field | Description |
|-------|-------------|
| **Username** | ClickHouse user | `default` |
| **Password** | User password (empty by default for local installs) |
| **Database** | Database to connect to | `default` |

<Tip>
ClickHouse ships with a `default` user that has no password on fresh installs. For production servers, always set a password or configure access control.
</Tip>

## HTTP Interface

TablePro connects to ClickHouse through its HTTP API, not the native TCP protocol (port 9000). The HTTP interface is enabled by default on port 8123 and supports all SQL operations.

If your ClickHouse server uses HTTPS, enable SSL/TLS in the connection form. The default HTTPS port is 8443.

<Note>
Some ClickHouse cloud providers only expose the HTTP(S) interface. TablePro works with these out of the box since it uses HTTP natively.
</Note>

## Example Configurations

### Local Development Server

```
Name:     Local ClickHouse
Host:     localhost
Port:     8123
Username: default
Password: (empty)
Database: default
```

### Docker ClickHouse Container

Start a ClickHouse container for local testing:

```bash
docker run -d \
  --name clickhouse \
  -p 8123:8123 \
  -p 9000:9000 \
  -e CLICKHOUSE_USER=default \
  -e CLICKHOUSE_PASSWORD=secret \
  clickhouse/clickhouse-server:latest
```

Then connect with:

```
Name:     Docker ClickHouse
Host:     localhost
Port:     8123
Username: default
Password: secret
Database: default
```

### ClickHouse Cloud

```
Name:     ClickHouse Cloud
Host:     abc123.us-east-1.aws.clickhouse.cloud
Port:     8443
Username: default
Password: (your cloud password)
Database: default
```

<Note>
ClickHouse Cloud requires HTTPS (port 8443). Enable SSL/TLS in the connection form.
</Note>

### Remote Server

```
Name:     Production ClickHouse
Host:     clickhouse.example.com
Port:     8123
Username: app_user
Password: (secure password)
Database: analytics
```

<Warning>
For production ClickHouse instances, consider using [SSH tunneling](/databases/ssh-tunneling) if the HTTP port is not encrypted.
</Warning>

## SSL/TLS Connections

TablePro supports SSL/TLS encryption for ClickHouse connections. Configure SSL in the **SSL/TLS** section of the connection form.

| SSL Mode | Description |
|----------|-------------|
| **Disabled** | No SSL encryption (HTTP on port 8123) |
| **Required** | Require SSL encryption (HTTPS, typically port 8443) |

<Tip>
ClickHouse Cloud and most managed ClickHouse services require HTTPS. Set the port to 8443 and enable SSL.
</Tip>

## SSH Tunnel Support

You can connect to ClickHouse through an SSH tunnel for secure access to remote servers. See [SSH Tunneling](/databases/ssh-tunneling) for detailed setup instructions.

## Features

### Database Browsing

After connecting, the sidebar lists all databases on the server. Expand a database to see its tables and views. Switch databases using the database switcher in the toolbar.

### Table Browsing

For each table, TablePro shows:

- **Structure**: Columns with ClickHouse data types, default expressions, and comments
- **Indexes**: Data skipping indices (minmax, set, bloom_filter, etc.)
- **DDL**: The full CREATE TABLE statement including engine and settings
- **Parts**: Partition and part details from `system.parts` (rows, disk size, active status)

### Engine Information

ClickHouse tables use storage engines that determine how data is stored and queried. TablePro displays the engine type and its parameters in the table structure view. Common engines:

| Engine | Use Case |
|--------|----------|
| **MergeTree** | Primary engine for production analytics tables |
| **ReplacingMergeTree** | Deduplication by sorting key |
| **SummingMergeTree** | Pre-aggregation of numeric columns |
| **AggregatingMergeTree** | Incremental aggregation with AggregateFunction columns |
| **Log / TinyLog** | Small tables, development, and testing |
| **Memory** | In-memory tables for temporary data |
| **Distributed** | Queries across shards in a cluster |

### Sorting Keys and Partition Keys

ClickHouse tables are ordered by their sorting key (ORDER BY clause in the table definition). TablePro displays the sorting key, partition key, and primary key in the table structure view. These are critical for query performance since ClickHouse reads data in sorted order.

### Data Skipping Indices

ClickHouse supports secondary indices that skip granules during queries. TablePro shows these in the Indexes tab:

```sql
-- Example: bloom filter index on a URL column
ALTER TABLE hits ADD INDEX url_idx url TYPE bloom_filter GRANULARITY 4;
```

### Query Progress Tracking

During query execution, the toolbar shows live progress: rows read and bytes processed. After completion, a summary displays total execution time, rows read, and bytes read. This data is polled from `system.processes`.

### EXPLAIN Variants

ClickHouse supports multiple EXPLAIN modes. Click the Explain dropdown in the query editor to choose:

| Variant | Description |
|---------|-------------|
| **Plan** | Logical query plan (default) |
| **Pipeline** | Physical execution pipeline with thread/port info |
| **AST** | Abstract syntax tree of the parsed query |
| **Syntax** | Query after syntax optimizations |
| **Estimate** | Estimated rows, marks, and parts to read |

### Server-side Query Cancellation

When you cancel a running query, TablePro sends a `KILL QUERY` command to the ClickHouse server. This stops the query on the server side, not just the HTTP connection.

### Query Editor

Write and execute ClickHouse SQL queries in the editor:

```sql
-- Aggregation query
SELECT
    toDate(event_time) AS date,
    count() AS events,
    uniqExact(user_id) AS unique_users
FROM analytics.events
WHERE event_time >= now() - INTERVAL 7 DAY
GROUP BY date
ORDER BY date;

-- System tables
SELECT name, engine, total_rows, total_bytes
FROM system.tables
WHERE database = 'analytics';

-- Table partitions
SELECT partition, sum(rows) AS rows, formatReadableSize(sum(bytes_on_disk)) AS size
FROM system.parts
WHERE table = 'events' AND active
GROUP BY partition
ORDER BY partition;
```

### Data Editing

TablePro supports editing cell values, inserting rows, and deleting rows in ClickHouse tables. Edits are submitted as standard INSERT, ALTER TABLE UPDATE, and ALTER TABLE DELETE statements.

<Warning>
ClickHouse mutations (UPDATE and DELETE) are asynchronous. They execute in the background and may take time to complete on large tables. See the Limitations section for details.
</Warning>

### Export and Import

Export query results or table data to CSV, JSON, and other formats. Import data from CSV files into ClickHouse tables.

## System Databases

ClickHouse includes built-in system databases:

| Database | Description |
|----------|-------------|
| **system** | Server metrics, query logs, parts info, cluster state |
| **information_schema** | SQL-standard metadata views (tables, columns, schemata) |
| **INFORMATION_SCHEMA** | Alias for `information_schema` (case-sensitive variant) |

The `system` database is particularly useful for monitoring:

```sql
-- Recent queries
SELECT query, read_rows, elapsed, memory_usage
FROM system.query_log
WHERE type = 'QueryFinish'
ORDER BY event_time DESC
LIMIT 20;

-- Disk usage by table
SELECT database, table, formatReadableSize(sum(bytes_on_disk)) AS size
FROM system.parts
WHERE active
GROUP BY database, table
ORDER BY sum(bytes_on_disk) DESC;
```

## Troubleshooting

### Connection Refused

**Symptoms**: "Connection refused" or timeout

**Causes and Solutions**:

1. **ClickHouse not running or HTTP interface disabled**
   ```bash
   # Check if ClickHouse is listening on port 8123
   curl http://localhost:8123/ping
   # Expected response: "Ok."
   ```

2. **Wrong port**
   - HTTP interface: 8123 (default)
   - HTTPS interface: 8443
   - Native TCP: 9000 (not used by TablePro)

3. **HTTP interface disabled in config**
   - Check `<http_port>` in `/etc/clickhouse-server/config.xml`

### Authentication Failed

**Symptoms**: "Authentication failed" or HTTP 403

**Solutions**:

1. Verify username and password
2. Check user exists in ClickHouse:
   ```sql
   SELECT name, auth_type FROM system.users;
   ```
3. Verify the user has access to the target database:
   ```sql
   SHOW GRANTS FOR app_user;
   ```

### Connection Timeout

**Symptoms**: Connection hangs or times out

**Solutions**:

1. Verify host and port are correct
2. Check network connectivity and firewall rules
3. For cloud-hosted ClickHouse, ensure your IP is in the allowed list

## Known Limitations

- **No foreign keys.** ClickHouse does not support foreign key constraints. Referential integrity must be managed at the application level.
- **No transactions.** ClickHouse has no multi-statement transaction support. Each statement executes independently.
- **Asynchronous mutations.** UPDATE and DELETE operations run as background mutations via `ALTER TABLE UPDATE` and `ALTER TABLE DELETE`. They are not instant: on large tables, mutations may take seconds to minutes. Check mutation progress with:
  ```sql
  SELECT * FROM system.mutations WHERE is_done = 0;
  ```
- **No auto-increment.** ClickHouse does not support auto-increment columns.
- **Primary key is immutable.** The primary key and sorting key cannot be changed after table creation.
- **INSERT-optimized.** ClickHouse is designed for batch inserts, not single-row writes. Inserting one row at a time works but is not efficient for bulk loading.
